# P0.1 Security Change Log

## Scope

This release removes the insecure email-only mock Google login path and enforces server-side administrator authorization for NGO administration and FAQ mutations.

## Breaking security changes

- `POST /api/auth/google-login` has been removed. It never verified a Google credential and could authenticate any supplied email address.
- `GET /api/ngos/approve/:id` and `DELETE /api/ngos/reject/:id` have been removed.
- NGO verification now uses `PATCH /api/ngos/{id}/verification` and requires an authenticated `Admin` role.
- `POST /api/faq` and `DELETE /api/faq/{id}` now require an authenticated `Admin` role. `GET /api/faq` remains public.

## Compatibility

- Email/password login, registration, password reset, public FAQ reads, and legacy `Donar` role normalization remain available.
- Existing pending NGOs without `verificationStatus` remain visible in the pending-review queue when `isVerified` is false.

## Data migration

No blocking migration is required. The user schema supplies `verificationStatus: pending` for new records; legacy records are interpreted from `isVerified` at read time. A recommended operational migration is:

```js
db.users.updateMany(
  { role: 'NGO', verificationStatus: { $exists: false }, isVerified: true },
  { $set: { verificationStatus: 'approved' } }
)
db.users.updateMany(
  { role: 'NGO', verificationStatus: { $exists: false }, isVerified: false },
  { $set: { verificationStatus: 'pending' } }
)
```

## Verification

Run from the repository root:

```bash
(cd Backend && npm test)
(cd Frontend && npm run typecheck && npm run lint && npm run build)
```

The backend authorization suite verifies that Google login is no longer registered, anonymous administrative requests are denied, non-admin users are forbidden, and administrator role checks succeed.
